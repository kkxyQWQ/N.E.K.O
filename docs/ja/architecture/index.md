# アーキテクチャ概要

Project N.E.K.O. は、4つの独立したサーバーがWebSocket、HTTP、ZeroMQメッセージングを通じて連携し、統一ランチャーによってオーケストレーションされる**マルチプロセスマイクロサービスシステム**として構築されています。

## 4サーバー構成

| サーバー | ポート | エントリーポイント | 役割 |
|---------|--------|-------------------|------|
| **メインサーバー** | 48911 | `main_server.py` | Web UI、REST API、WebSocketチャット、TTS |
| **メモリサーバー** | 48912 | `memory_server.py` | セマンティック検索、時間インデックス付き履歴、メモリ圧縮、設定管理 |
| **モニターサーバー** | 48913 | `monitor.py` | ヘルスチェック、ランチャー連携用の設定エンドポイント |
| **エージェントサーバー** | 48915 | `agent_server.py` | バックグラウンドタスク実行（MCP、Computer Use、Browser Use、User Plugin） |

メインサーバーはユーザー向けのエントリーポイントです。Web UIの提供、すべてのREST APIリクエストの処理、リアルタイム音声/テキストチャットのためのWebSocket接続の維持を行います。メモリ、エージェント、モニターサーバーは内部サービスであり、メインサーバーがHTTP、ZeroMQ、WebSocketを通じてそれぞれと通信します。

## 通信パターン

```
┌──────────────────────────────────────────┐
│              Main Server (:48911)         │
│                                          │
│  FastAPI ─── REST Routers（11ルーター）   │
│  WebSocket ─── LLMSessionManager         │
│  ZeroMQ PUB ───┐                         │
│  ZeroMQ PULL ──┼── AgentEventBridge      │
│  HTTP Client ──┤                         │
└────────────────┼─────────────────────────┘
                 │
     ┌───────────┼───────────┐
     │           │           │
     ▼           ▼           ▼
  Memory      Agent      Monitor
  Server      Server     Server
  (:48912)    (:48915)   (:48913)
```

- **Main ↔ Memory**: メモリの保存/クエリ、設定抽出、メモリレビュー用HTTPリクエスト
- **Main ↔ Agent**: タスク委任と結果ストリーミング用ZeroMQ pub/sub、フラグと能力用HTTPプロキシ
- **Main ↔ Monitor**: ヘルスチェック、設定状態、ランチャー連携用HTTP
- **Launcher → All**: プロセス管理、ポートフォールバック、グレースフルシャットダウン

## 主要なアーキテクチャパターン

### ホットスワップセッション

`LLMSessionManager` は、現在のセッションがまだアクティブな間に、バックグラウンドで新しいLLMセッションを準備します。約40秒のサイクルがセッションのライフタイムを管理 — 閾値を過ぎるとメモリがアーカイブされ、新しいセッションがバックグラウンドで事前ウォームアップされます。現在のターンが終了すると、事前にウォームアップされたセッションにゼロダウンタイムでシームレスに切り替わります。移行中の音声はキャッシュされ、レート制限付きでフラッシュされます。

### キャラクターごとの分離

各キャラクター（`lanlan_name` で識別）は、それぞれ独自の以下を持ちます：
- `LLMSessionManager` インスタンス
- 同期コネクタースレッド
- WebSocketロック
- メッセージキュー
- シャットダウンイベント

キャラクターは再起動なしでランタイム時に追加・削除できます。既存のアクティブセッションは保持されたまま、設定がホットリロードされます。

### 非同期/同期の境界

FastAPIハンドラーは非同期です。TTS合成はキューベースの通信を使用する専用スレッドで実行されます。音声処理は `soxr` ステートフルリサンプラー（24kHz → 48kHz）を使用します。ZeroMQイベントブリッジはバックグラウンド受信スレッドをNOBLOCK送信とACK/リトライメカニズムで実行します。

### スタートアップロック

ランチャーはプラットフォーム固有のロック（Windows Named Mutex `Global\NEKO_LAUNCHER_STARTUP_LOCK`、POSIX file lock）を使用して、重複ランチャーインスタンスを防止します。

### ヘルスチェックシグネチャ

すべてのサーバーはヘルスエンドポイントで `{"app": "N.E.K.O", "service": "..."}` を返します。ランチャーはこのシグネチャを使用して、同じポートを占有する他のサービスからN.E.K.O.プロセスを識別します。
